Техническое задание: Система внутренних переводов монет (P2P-транзакции)

1. Цель и назначение
   Разработать функционал, позволяющий пользователям приложения переводить заработанные монеты (Tap-to-earn) другим пользователям внутри экосистемы. Система должна включать пользовательский интерфейс поиска получателя, подтверждения перевода, историю операций, а также административную панель для мониторинга и управления транзакциями.

2. Общая архитектура и логика работы
   • Внутренняя валюта: «Монеты» (целочисленный тип данных, без копеек).
   • Идентификатор: Уникальный telegram_id пользователя (получаемый через WebApp initData).
   • Поиск: По nickname (username в Telegram, без символа @) или telegram_id (на случай, если у пользователя нет username).
   • Целостность: Все транзакции должны быть атомарными: списание у отправителя происходит только при успешном зачислении получателю. При любой ошибке — откат транзакции (rollback).

3. Функциональные требования (Пользовательская часть)
   3.1. Доступ к разделу
   На главном экране (интерфейс тапов) добавить кнопку «Кошелек» или «Переводы».
   • Визуал: Иконка бумажника/стрелки.
   • Действие: Открывает главный экран раздела переводов.
   3.2. Экран: Отправка перевода (Переводы / Отправить)
   Экран должен состоять из нескольких логических шагов или полей на одной странице.
   Шаг 1: Поиск получателя
   • Поле ввода: «Введите username или Telegram ID получателя».
   • Кнопка «Найти» (или автоматический поиск по мере ввода с задержкой debounce).
   • Результат поиска:
   • При успехе: отображается аватар (если есть), имя пользователя (first name / last name) и его username. Кнопка «Выбрать».
   • При ошибке: «Пользователь не найден» (пользователь не начинал взаимодействие с ботом/приложением ИЛИ в базе нет такого user_id).
   • Валидация: Запрещено переводить самому себе (кнопка «Выбрать» неактивна или скрыта, если найденный ID совпадает с ID отправителя).
   Шаг 2: Ввод суммы
   • После выбора пользователя открывается интерфейс ввода суммы.
   • Показывается текущий баланс отправителя (например, «Доступно: 10 500 монет»).
   • Поле ввода: Только цифры (Input[type=number]).
   • Кнопки быстрого ввода: (Опционально) кнопки «+100», «+500», «Всё».
   • Валидация:
   • Сумма должна быть больше нуля.
   • Сумма не должна превышать текущий баланс.
   Шаг 3: Подтверждение (Экран-Чек)Показывается сводка:
   • Кому: Имя и username получателя.
   • Сумма: X монет.
   • Комиссия: 0% (или указать комиссию, если планируется в будущем).
   • К оплате: X монет.
   • Кнопки: «Подтвердить перевод» и «Назад».
   Шаг 4: Результат
   • Успех: Всплывающее окно (alert) или отдельный экран с галочкой и сообщением «Переведено X монет пользователю @username». Кнопка «Вернуться в кошелек».
   • Ошибка: «Недостаточно средств» или «Ошибка сети. Попробуйте снова».
   3.3. Экран: История переводов
   На главном экране раздела переводов добавить вкладку или кнопку «История».
   Требования к отображению:
   • Лента операций (пагинация или бесконечная лента).
   • Каждый элемент должен содержать:
   • Тип операции: Исходящий перевод (минус, красный) или Входящий перевод (плюс, зеленый).
   • С кем операция: Имя и username контрагента.
   • Сумма.
   • Дата и время (в формате DD.MM.YYYY HH:MM).
   • Статус: «Выполнен» (успешно).
   • (Опционально) Комментарий, если функция комментариев будет добавлена позже.
   • При клике на элемент истории — открывается детализация транзакции (ID транзакции, полная дата, точная сумма, комиссия, статус).

4. Функциональные требования (Административная панель)
   Админка должна быть доступна по отдельной ссылке или внутри бота для пользователей с ролью admin.
   4.1. Дашборд (Главная)
   • Общее количество проведенных транзакций (за все время / за сегодня).
   • Общий объем переведенных монет (за период).
   • Количество уникальных пользователей, совершавших переводы (отправителей).
   4.2. Управление и мониторинг транзакций
   Таблица со списком всех переводов.
   • Колонки:
   • ID транзакции (внутренний номер).
   • Дата и время.
   • Отправитель (Telegram ID, Username, Имя).
   • Получатель (Telegram ID, Username, Имя).
   • Сумма.
   • Баланс отправителя ДО операции.
   • Баланс отправителя ПОСЛЕ операции.
   • Статус (Успешно / Ошибка / Отмена).
   • Фильтры:
   • По ID пользователя (отправитель/получатель).
   • По дате (календарь).
   • По сумме (больше/меньше).
   • По статусу.
   • Поиск: Поиск по Username/ID.
   4.3. Управление балансами пользователей (Failsafe)
   • Просмотр баланса: Вбить Telegram ID пользователя — увидеть текущий баланс и историю его транзакций.
   • Ручная корректировка баланса (начисление/списание): Функция для поддержки и исправления ошибок.
   • Поле: ID пользователя.
   • Поле: Сумма (+ начислить, - списать).
   • Поле: Причина (обязательно для аудита).
   • Кнопка: «Исполнить». Все ручные операции также должны логироваться в отдельной таблице (журнал действий админа).
   4.4. Логирование действий администратора
   Ведение отдельного лога (кто из админов, когда и зачем изменил баланс пользователя или настройки системы).

5. Детали реализации (Backend & Database)
   5.1. Модели данных (Таблицы в БД)
   Таблица users:
   • id (PK)
   • telegram_id (bigint, unique index)
   • username (varchar, индекс для поиска)
   • first_name, last_name
   • balance (bigint, по умолчанию 0)
   • created_at
   Таблица transactions:
   • id (PK) — уникальный номер транзакции.
   • sender_id (FK to users.id, индекс) — кто отправил.
   • receiver_id (FK to users.id, индекс) — кто получил.
   • amount (bigint) — сумма перевода (целое число).
   • sender_balance_before (bigint) — баланс отправителя до.
   • sender_balance_after (bigint) — баланс отправителя после.
   • receiver_balance_before (bigint) — баланс получателя до.
   • receiver_balance_after (bigint) — баланс получателя после.
   • status (enum: 'completed', 'failed', 'pending').
   • created_at (timestamp).
   • completed_at (timestamp).
   Таблица admin_logs:
   • id
   • admin_id (FK to users.id)
   • action (text, например: "Изменил баланс user_id 123 на +500. Причина: Ошибка начисления")
   • created_at
   5.2. API Эндпоинты (Backend)
   • POST /api/transfer/search — поиск пользователя по username/ID.
   • POST /api/transfer/send — выполнение перевода.
   • Вход: sender_id (из контекста авторизации), receiver_id, amount.
   • Логика: Проверка баланса -> Начало транзакции БД -> Обновление баланса отправителя (balance -= amount) -> Обновление баланса получателя (balance += amount) -> Запись в таблицу transactions -> Commit.
   • GET /api/transfer/history — получение истории для конкретного пользователя (с пагинацией).
   • Админские:
   • GET /api/admin/transactions (с фильтрацией и пагинацией).
   • POST /api/admin/user/balance (ручное изменение баланса, требует авторизацию админа и логирование).

6. Безопасность и защита от накруток
   • Инициализация данных: Все запросы к API должны проверять подпись initData, переданную от Telegram WebApp, чтобы убедиться, что запрос идет от реального пользователя, а не от злоумышленника.
   • Защита от двойной траты (Race Conditions): Использовать блокировки (row-level locking в SQL) или атомарные операции обновления (UPDATE users SET balance = balance - amount WHERE id = X AND balance >= amount). Проверять количество затронутых строк.
   • Лимиты:
   • Ввести ограничение на максимальную сумму одной транзакции (например, не больше 1 000 000 монет за раз).
   • Ввести ограничение на количество транзакций в минуту с одного аккаунта (Rate Limiting).
   • Валидация на бэкенде: Никогда не доверять данным, пришедшим с клиента. Сумма, ID получателя и факт наличия средств должны перепроверяться на сервере.

7. UI/UX требования к мини-приложению
   • Адаптивный дизайн под мобильные устройства.
   • Плавные анимации (нажатие кнопок, появление/исчезновение элементов).
   • Понятные подсказки (Placeholder'ы, сообщения об ошибках рядом с полем ввода, а не в алертах сверху).
   • Состояния загрузки (лоадеры) при поиске пользователя и отправке перевода.
   • Цветовое кодирование: Зеленый (приход), Красный (расход), Синий (нейтральный, ссылки).

8. Критерии приемки (Definition of Done)
   • Пользователь может найти другого пользователя по username.
   • Нельзя перевести больше, чем есть на балансе.
   • Нельзя перевести самому себе.
   • После подтверждения перевода баланс отправителя уменьшается, баланс получателя увеличивается мгновенно.
   • Транзакция отображается в истории у обоих пользователей с правильным знаком (+/-).
   • В админ-панели видна эта транзакция, видны балансы до и после.
   • Администратор может вручную скорректировать баланс (начислить/списать монеты), и это действие логируется.
   • Система выдерживает 50 одновременных запросов на перевод без ошибок (проверка Race Condition).
